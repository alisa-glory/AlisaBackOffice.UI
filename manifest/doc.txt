Flow+Logic ให้ Admin เทรน
1.เวลามีคำถามใหม่ ที่ Status ของ Davinci ตัวแรก บอกว่าเป็น Update มา ให้ส่งเข้ามาให้ Admin เห็น โดย Admin จะไม่เห็นคำถามซ้ำในแต่ละคำถามที่ตรงกันเป๊ะๆ จะเห็นแค่ครั้งเดียวที่เก่าที่สุด
(แต่ Super Admin จะเห็นหน้าที่ Admin เลือกมา และ ยอมรับ/ไม่ยอมรับได้ หรือดู Status เพิ่มเติมได้ โดยเห็นก่อน-หลังแก้ไข หลังจาก Approve แล้ว จะขึ้นข้อมูลที่เป็นปัจจุบัน)

*ของข้อ2ใหญ่นี้ ถ้ามีการอัพเดท Database เรา (Embed) ควรใส่วันที่ Approve ไปด้วย (อาจจะใช้ช่อง Completion date เผื่อใช้ในการให้น้ำหนักข้อมูลมากกว่า กรณีคำถามใน Embed ซ้ำกัน)
 2.Admin จะเลือกว่าคำถามนี้อยู่ประเภทไหน ระหว่าง 
--1.Fix = คำถาม และ Database เรา (Embed) ok แล้ว ไม่ต้องทำอะไรเพิ่ม ไม่ว่าเวลาจะผ่านไปนานแค่ไหนถ้าถามแบบนี้ก็จะตอบแบบนี้  ไม่ต้องอัพเดทอะไรอีก
กระบวนการทำงาน: ไม่เกิดอะไรขึ้น แค่ถ้ามีคำถามนี้เข้ามาอีกก็ไม่เจอว่าต้องมาเลือก Status อีกเพราะถูก Fix ไว้แล้ว

--2.Update from internet = Database เรา (Embed) ไม่โอเค แต่ข้อมูลจาก Internet ok และ เป็นข้อมูลที่ไม่ว่าจะนานแค่ไหน ถ้าถามแบบนี้ก็จะตอบแบบ Internet รอบนี้  ไม่ต้องอัพเดทอะไรอีก
กระบวนการ: เพิ่มข้อมูลใน Database เรา (Embed) โดยคำถามอิงจากที่ลูกค้าถาม (เป็นอังกฤษ) และคำตอบ อิงจากการตอบของ Internet ที่ Admin เห็นในครั้งนั้น และถ้ามีคำถามนี้เข้ามาอีกก็ไม่เจอว่าต้องมาเลือก Status อีกเพราะถูก Update ไว้แล้ว

--3.Update from human =  Database เรา (Embed) ไม่โอเค และข้อมูลจาก Internet ก็ไม่โอเค ดังนั้น Admin(คน) เลยต้องไปหาคำตอบที่เหมาะสมมากรอก ทั้งนี้เป็นข้อมูลที่ไม่ว่าจะนานแค่ไหน ถ้าถามแบบนี้ก็จะตอบแบบ รอบนี้  ไม่ต้องอัพเดทอะไรใหม่อีก
กระบวนการ: เพิ่มข้อมูลใน Database เรา (Embed) โดยคำถามอิงจากที่ลูกค้าถาม (เป็นอังกฤษ) และคำตอบ อิงจากการตอบของ Admin(คน) ที่กรอกในครั้งนี้ และถ้ามีคำถามนี้เข้ามาอีกก็ไม่เจอว่าต้องมาเลือก Status อีกเพราะถูก Update ไว้แล้ว

--4.Update Everyday =  ไม่เกี่ยวกับว่า Embed โอเคแล้วหรือไม่ แต่ ข้อมูลจากอินเตอร์เน็ตต้องโอเค และที่สำคัญคือ Admin ต้องรู้ว่าคำถามนี้มันควรต้องคอยมีการอัพเดทเป็นระยะ เป็นคำถามที่ถ้า Fix คำตอบแล้วอนาคตมันจะตอบผิด เช่น คำถามที่เกี่ยวกับ เรื่องราว "ล่าสุด" "ใหม่สุด" "เมื่อวาน" "วันนี้" "เดือนก่อน" "ปีก่อน" เป็นต้น(อาจจะมีอย่างอื่นอีก Admin ต้องใช้ Logic) เช่น ไอโฟนรุ่นล่าสุด ผลบอลแมนยูเมื่อวานนี้ หวยล่าสุด ผลฟุตบอลโลกรอบล่าสุด
กระบวนการ: เจอครั้งแรกกระบวนการเหมือน Update from internet คือ เพิ่มข้อมูลใน Database เรา (Embed) โดยคำถามอิงจากที่ลูกค้าถาม (เป็นอังกฤษ) และคำตอบ อิงจากการตอบของ Internet ที่ Admin เห็นในครั้งนั้น และถ้ามีคำถามนี้เข้ามาอีก Admin ก็ไม่เจอว่าต้องมาเลือก Status อีกเพราะถูก Update Everydayไว้แล้ว แต่ฝั่ง Server จะต้องเอาข้อมูลของ Internet ไปอัพเดทใน Row ของคำถามเดิม ใน Database เรา (Embed) วันละครั้งที่มีการถามเข้ามาใหม่ เพื่อให้ข้อมูลตรงนี้ Update ตลอดทุกวัน
(ยกตัวอย่างเช่น ถ้าวันนึงไอโฟน 15 ออกมา แล้ว Status นี้ถูกติ้กไว้กับคำถามที่ว่า ไอโฟนรุ่นล่าสุดคือออะไร Database เราจะตอบ 14 แต่ Internet ตอบ 15 มันจะพลาดแค่ครั้งเดียว หลังจากนั้นวันนั้น Database เรามันก็จะตอบ 15 เพราะถูกตั้งให้คอยอัพเดทไว้วันละครั้ง)

--5.Bug = จริงๆแล้วเป็นข้อมูลที่ต้องอัพเดททุกวัน แต่ Internet ดันตอบผิด หรือ Internet ตอบแล้ว Toxic ให้ตีเป็น Bug เพื่อให้ Super Admin รีเช็ค ซึ่ง Admin จะกรอกเหมือน Update from human ไปก่อนก็ได้ (คล้ายๆตอบแก้ขัดไปก่อน) แต่ Status ก็จะขึ้นเป็น Bug ไป เพื่อมาคอยรีเช็คได้เรื่อยๆ เพราะหมายความว่า ข้อมูลจากอินเตอร์เน็ตไม่ควรนำมาใช้ในคำถามนี้ แต่ มนุษย์ตอบไป อาจจะถูกต้องวันนี้ แต่อนาคตอาจจะผิดได้ 
กระบวนการ: ถ้า Admin กรอกคำตอบได้ ก็จะเหมือน Update from human แตกต่างแค่ Status จะเป็น Bug และต้องคอยมาตรวจสอบอยู่เรื่อยๆ อนาคต Super admin อาจจะเปลี่ยนไปเป็น Status อื่นเช่น Update Everyday ก็ได้หากทดสอบแล้วว่าอินเตอร์เน็ตตอบ ok แล้ว


ตาราง Logic ครับ + คิดว่าทำเป็นช่องๆให้ Admin ติ๊กเลือก ทีละช่อง น่าจะดีกว่า (คือ Admin เลือกที่ Status เลยไม่ได้ Status ถูกเลือกโดยระบบLogic ที่ผูกไว้ Admin มีหน้าที่ติ๊กตอบในแต่ละคำถามครับ)
